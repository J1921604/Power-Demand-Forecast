# Power-Demand-Forecast プロジェクト憲法

## 核心原則 (Core Principles)

### I. テスト駆動開発の徹底 (TDD優先)

**原則**:
- すべての機能実装において、テストを先に作成し、仕様に対する検証を必須とする
- テストが失敗することを確認してから実装を開始する（Red-Green-Refactorサイクル）
- 単体テスト、統合テスト、契約テストを段階的に実施する
- テストカバレッジは最低限の品質保証ではなく、仕様の完全性証明として扱う

**理由**:
- 仕様の誤解を早期に発見し、手戻りコストを最小化する
- リファクタリング時の安全性を保証し、技術的負債を防ぐ
- ドキュメントとしても機能し、コードの意図を明確にする
- 電力需要予測という重要インフラにおいて、予測精度と安定性を保証する

**適用例**:
- モデル学習スクリプトでは、期待精度（R² > 0.80）を先にテストケースとして定義
- データ処理パイプラインでは、入力/出力の型と範囲を先に検証
- API連携では、Open-Meteo APIのレスポンス形式を契約テストで検証

---

### II. セキュリティファースト原則

**原則**:
- セキュリティ要件を機能要件より優先する
- 機密データ（APIキー、認証トークン、個人情報）の平文保存を禁止
- すべての外部API通信はHTTPS/TLSを必須とする
- 環境変数による機密情報管理を徹底する
- セキュリティパッチは最優先で適用する

**理由**:
- 後からのセキュリティ対策は設計変更とコスト増大を伴う
- データ漏洩は事業継続性に致命的な影響を与える
- 電力需要データは社会インフラの機密情報である可能性がある
- 信頼性とコンプライアンスを維持する

**適用例**:
- GitHub ActionsではSecretsを使用し、環境変数でAPI情報を管理
- ローカル環境では`.env`ファイル（`.gitignore`で除外）を使用
- Open-Meteo API通信は必ずHTTPSエンドポイントを使用
- パスワード、トークンは必ずハッシュ化または暗号化して保存

---

### III. パフォーマンス定量化原則

**原則**:
- パフォーマンス閾値を定量化し、受入基準に組み込む
- すべてのモデル学習・予測処理に実行時間とメモリ使用量の上限を設定
- 精度指標（RMSE、R²、MAE）を明確に定義し、閾値未達時は自動アラート
- GitHub Actions実行時間の制限を考慮した最適化を行う

**理由**:
- 曖昧な「高速化」ではなく、測定可能な目標を設定することで改善が可能になる
- 本番環境（GitHub Actions）での実行可能性を保証する
- ユーザー体験（ダッシュボードの応答速度）を維持する
- リソースコスト（CI/CD実行時間）を最適化する

**適用例**:
- LightGBM学習時間: < 30秒（GitHub Actions環境）
- Keras学習時間: < 60秒（GitHub Actions環境）
- ダッシュボードAPI応答時間: < 2秒（ローカルHTTPサーバー）
- R²スコア: > 0.80（閾値未達時にGitHub Issue自動作成）
- メモリ使用量: float32型を使用し、メモリ使用量を約50%削減

---

### IV. データ保護の徹底

**原則**:
- 機密データの平文保存を禁止し、暗号化またはハッシュ化を必須とする
- データアクセスログを記録し、監査可能性を確保する
- 個人情報を含む可能性のあるデータは匿名化処理を実施
- データのバックアップと復旧手順を文書化する

**理由**:
- データ漏洩による法的リスクとブランド毀損を防ぐ
- GDPR、個人情報保護法などのコンプライアンス要件に対応
- データの完全性と可用性を保証する
- 電力需要データの機密性を維持する

**適用例**:
- 気温データはOpen-Meteo APIから取得し、公開データのみを使用
- ローカルストレージにはセッション情報のみ保存（個人情報なし）
- サーバーログには個人を特定できる情報を含めない
- データベース使用時は暗号化ストレージを採用

---

### V. 依存関係の再現性確保

**原則**:
- 外部依存パッケージはバージョンを固定し、`requirements.txt`で厳密に管理
- Python実行環境のバージョンを明記する（Python 3.10.11）
- GitHub Actionsのアクションバージョンもタグで固定する
- 依存パッケージの更新は計画的に行い、影響範囲を評価する

**理由**:
- 環境差異による予期しない動作を防ぐ
- 機械学習モデルの再現性を保証する
- デプロイ後のトラブルシューティングを容易にする
- CI/CD環境とローカル環境の一貫性を維持する

**適用例**:
- `requirements.txt`ですべてのパッケージを`==`で固定（例: `lightgbm==3.3.5`）
- GitHub Actionsで`python-version: '3.10.11'`を明記
- GitHub Actionsアクションは`@v4`などのメジャーバージョンタグを使用
- 依存パッケージ更新時は必ずテストを実行し、精度劣化がないことを確認

---

### VI. 仕様実装乖離の検出

**原則**:
- 仕様と実装の乖離をレビューで検知・是正する
- すべてのPull Requestに仕様へのリンクを含める
- 仕様変更時は影響を受けるコードを特定し、同時更新する
- ドキュメント（README、仕様書）とコードの整合性を定期的に検証する

**理由**:
- 仕様と実装の乖離は品質低下とメンテナンス性悪化の原因となる
- 長期的なプロジェクト運用において、ドキュメントが実態を反映しなくなる問題を防ぐ
- コードレビューの効率を向上させる
- チーム間のコミュニケーションコストを削減する

**適用例**:
- Pull Requestテンプレートに「関連仕様」セクションを追加
- 仕様書（`docs/完全仕様書.md`）とREADMEの整合性を定期チェック
- GitHub Actionsワークフローの変更時は`docs/DEPLOY_GUIDE.md`を同時更新
- モデル精度閾値の変更時は、仕様書とコードの両方を更新

## ブランチ戦略

### 仕様ブランチ（mainブランチから派生）

```bash
git checkout main
git checkout -b <番号>-<短い名前>
# 例: git checkout -b 001-Power-Demand-Forecast
```

**用途**:
- 新機能の仕様策定
- アーキテクチャ設計
- ドキュメント作成

**マージ条件**:
- 仕様レビュー完了
- 関連ドキュメント更新完了

---

### 実装ブランチ（仕様ブランチから派生）

```bash
git checkout 001-Power-Demand-Forecast
git checkout -b feature/impl-001-Power-Demand-Forecast
```

**用途**:
- 実装作業
- テスト作成・実行
- コードレビュー

**マージ条件**:
- すべてのテスト通過
- コードレビュー承認
- パフォーマンス閾値達成
- セキュリティチェック完了

## 開発ワークフロー

### 作業順序（厳守）

```mermaid
flowchart LR
    A[憲法確認] --> B[仕様作成]
    B --> C[実装計画]
    C --> D[タスク分解]
    D --> E[テスト作成]
    E --> F[実装]
    F --> G[検証]
    G --> H[レビュー]
    H --> I[マージ]
```

1. **憲法確認**: 本憲法の原則を確認し、準拠を確認
2. **仕様作成**: 機能仕様書を作成（`specs/<番号>-<機能名>/spec.md`）
3. **実装計画**: アーキテクチャと技術選定を文書化（`specs/<番号>-<機能名>/plan.md`）
4. **タスク分解**: 実装タスクをチケット化（`specs/<番号>-<機能名>/tasks.md`）
5. **テスト作成**: TDDに従い、先にテストを作成
6. **実装**: テストが通るように実装
7. **検証**: すべてのテストと品質基準を確認
8. **レビュー**: Pull Requestを作成し、レビュー承認を得る
9. **マージ**: mainブランチにマージ

---

### フロントエンドとバックエンドの同時起動

**開発方針**:
- フロントエンド（ダッシュボード）とバックエンド（Python HTTPサーバー）を同時に起動するコードを生成
- 正常に動作するまで繰り返し検証し、エラー修正を完了する

**実装例**:

```bash
# HTTPサーバー起動（バックエンド）
cd AI
py -3.10 server.py
# ブラウザ自動起動（フロントエンド）: http://localhost:8002/
```

### Mermaid図v11準拠のベストプラクティス

**推奨事項**:
1. **gitGraph使用時の注意**:
   - 日本語を避けるか、flowchart/graph形式を使用
   - `tag:`構文は非推奨、代わりにノードで表現
   - `${{}}`などの特殊文字をノードラベルに使用できない

2. **日本語対応**:
   - flowchart、graph、sequenceDiagramは日本語完全対応
   - ノードラベル、エッジラベル、Noteで日本語使用可能
   - gitGraph構文は日本語コミットメッセージに対応していない

3. **推奨構文**:
   - ブランチ戦略: `flowchart TB + subgraph`
   - プロセスフロー: `flowchart TD/LR`
   - 時系列: `sequenceDiagram`
   - 状態遷移: `stateDiagram-v2`

## ガバナンス

### 憲法の優先順位

**原則**:
- この憲法はすべてのプロジェクト慣行より優先される
- 憲法と矛盾する慣行は即座に是正される
- 憲法の解釈に疑義がある場合は、プロジェクトリーダーが最終判断を行う

---

### 憲法の改訂手順

**プロセス**:
1. 改訂提案をGitHub Issueで起票
2. 影響範囲分析を実施
3. レビュー承認を取得
4. バージョン番号を更新（セマンティックバージョニング）
   - **MAJOR**: 後方互換性のない原則削除または再定義
   - **MINOR**: 新原則/セクション追加または重要な拡張
   - **PATCH**: 明確化、文言修正、非セマンティックな改善
5. 移行計画を文書化
6. 関連テンプレートとドキュメントを同期更新
7. 最終改訂日を更新

---

### 品質ゲート

**必須チェック項目**:
- [ ] すべてのテストが通過している
- [ ] パフォーマンス閾値を達成している
- [ ] セキュリティチェックが完了している
- [ ] ドキュメントが最新状態である
- [ ] 依存関係がバージョン固定されている
- [ ] 仕様と実装が一致している

**重大変更の定義**:
- 公開APIの変更
- データスキーマの変更
- 精度閾値の変更
- セキュリティポリシーの変更
- 依存パッケージのメジャーバージョンアップ

**重大変更の承認プロセス**:
1. 変更提案のドキュメント作成
2. 影響範囲の分析
3. レビュー承認（最低1名）
4. ステージング環境でのテスト
5. 本番適用

---

### コンプライアンスチェック

**レビュー項目**:
- すべてのPull Requestで憲法準拠を検証
- 複雑性の導入には明確な理由が必要
- テスト不足のコードはマージ拒否
- セキュリティ要件違反は即座に修正
